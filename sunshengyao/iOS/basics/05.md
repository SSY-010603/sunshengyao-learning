# 05 — 状态管理与数据绑定

> 本文覆盖公共概念：③ 状态管理与数据绑定

阶段 1 你学会了拼 UI、处理点击、做跳转。但一个真实 App 的页面不是静态的 —— 数据从服务端拿回来后要显示到 UI 上，用户修改了设置要同步到别的页面，列表数据变了要刷新。

核心问题：**数据变了，UI 怎么跟着变？**

---

## 一句话定义

**状态管理** = 维护 App 的数据状态 + 当数据变化时通知 UI 更新。
**数据绑定** = 数据和 UI 之间的自动同步机制。

---

## C++ 直觉

在 C++ 里你大概率写过类似的东西：

```cpp
// C++ Qt 的信号槽 —— 最经典的数据绑定
connect(model, &DataModel::countChanged, label, &QLabel::setText);
// data.count 变了 → 自动调用 label->setText

// 或者 MFC 的 Document-View 架构
// Document (数据) 变了 → 通知 View (UI) 重绘
```

iOS ObjC 没有一个"万能绑定框架"，但有三种机制拼起来用：

| 机制 | C++ 类比 | 适用场景 |
|------|----------|----------|
| KVO | Qt 信号槽（属性级观察） | 观察某个属性的变化 |
| NSNotification | 事件总线 / 全局广播 | 跨层级、跨页面的松耦合通知 |
| Delegate + 手动刷新 | 回调接口 + 手动调用 | 一对一的状态传递（最常用） |

---

## 三种机制详解

### 1. KVO — 属性级观察者

KVO（Key-Value Observing）是 ObjC 运行时内建的能力：对任意 NSObject 子类的属性，你可以在属性值变化时收到回调。

**C++ 类比**：Qt 的 `Q_PROPERTY` + `notify` 信号，但 ObjC 不需要你声明信号，运行时自动帮你 hook setter。

```objc
// === Model ===
@interface User : NSObject
@property (nonatomic, copy) NSString *name;
@property (nonatomic, assign) NSInteger score;
@end

// === Observer（某个 VC）===
- (void)viewDidLoad {
    [super viewDidLoad];
    
    // 注册观察：观察 user 的 "score" 属性
    [self.user addObserver:self
               forKeyPath:@"score"
                  options:NSKeyValueObservingOptionNew | NSKeyValueObservingOptionOld
                  context:nil];
}

// 属性变化时的回调 —— 必须实现这个方法
- (void)observeValueForKeyPath:(NSString *)keyPath
                      ofObject:(id)object
                        change:(NSDictionary<NSKeyValueChangeKey,id> *)change
                       context:(void *)context
{
    if ([keyPath isEqualToString:@"score"]) {
        NSInteger oldScore = [change[NSKeyValueChangeOldKey] integerValue];
        NSInteger newScore = [change[NSKeyValueChangeNewKey] integerValue];
        self.scoreLabel.text = [NSString stringWithFormat:@"得分：%ld", (long)newScore];
    }
}

// ⚠️ 必须移除，否则 crash（和 KVO 的实现机制有关）
- (void)dealloc {
    [self.user removeObserver:self forKeyPath:@"score"];
}
```

**KVO 底层原理**（面试常考，也帮你理解为什么必须 removeObserver）：

1. 注册 KVO 时，运行时**动态创建**一个 `NSKVONotifying_User` 子类
2. 该子类重写 `setScore:` 等被观察的 setter
3. 重写后的 setter 在调用原实现前后，通知观察者
4. 所以 KVO 只对 **通过 setter 修改** 的属性生效，直接改 ivar（`_score = 10`）不会触发

```objc
// 这会触发 KVO（走了 setter）
self.user.score = 100;

// 这不会触发 KVO（直接改 ivar）
_user.score = 100;  // ⚠️ 陷阱
```

**KVO 的坑**：

| 坑 | 说明 |
|----|------|
| 忘了 removeObserver → crash | 对象 dealloc 后，KVO 仍尝试向已释放的观察者发消息，EXC_BAD_ACCESS |
| 多个 keyPath 混在一个回调 | `observeValueForKeyPath:` 只有一个，所有观察都走这里，if-else 容易乱 |
| 直接改 ivar 不触发 | 必须 `self.xxx =` 走 setter |
| 线程不保证 | 在哪个线程改的值，回调就在哪个线程，UI 更新要自己 dispatch 回 main |

### 2. NSNotification — 全局广播

**C++ 类比**：一个全局的事件总线（`EventBus::subscribe("loginSuccess", callback)`），谁都能发，谁都能收，发收双方互不知道对方。

```objc
// === 发送方（比如登录成功后）===
- (void)loginSuccess {
    // 发通知，附带数据
    [[NSNotificationCenter defaultCenter] postNotificationName:@"UserLoginSuccess"
                                                        object:nil
                                                      userInfo:@{@"userId": @(12345)}];
}

// === 接收方（比如首页，想刷新用户信息）===
- (void)viewDidLoad {
    [super viewDidLoad];
    
    // 注册监听
    [[NSNotificationCenter defaultCenter] addObserver:self
                                             selector:@selector(onLoginSuccess:)
                                                 name:@"UserLoginSuccess"
                                               object:nil];
}

- (void)onLoginSuccess:(NSNotification *)note {
    NSDictionary *userInfo = note.userInfo;
    NSNumber *userId = userInfo[@"userId"];
    // 刷新 UI...
    [self reloadUserHeader];
}

// ⚠️ 同样必须移除
- (dealloc) {
    [[NSNotificationCenter defaultCenter] removeObserver:self];
}
```

**NSNotification vs KVO 对比**：

| | KVO | NSNotification |
|---|-----|----------------|
| 关系 | 观察者必须持有被观察对象 | 发收双方完全解耦 |
| 数据 | 自动拿到新旧值 | 自己塞 userInfo 字典 |
| 触发方式 | 属性 setter 自动触发 | 手动 post |
| 生命周期 | 跟着被观察对象 | 跟着 NSNotificationCenter（全局单例） |
| 典型场景 | 观察 model 的某个字段变化 | 全局事件：登录/登出、主题切换、语言切换 |

**NSNotification 的典型用例**（这些是你日常开发中最常见到的）：

```
UIApplicationDidEnterBackgroundNotification   — App 进入后台
UIApplicationWillEnterForegroundNotification  — App 回到前台
UITextFieldTextDidChangeNotification           — 文本框内容变化
AVPlayerItemDidPlayToEndTimeNotification      — 视频播放完毕
```

> 你不需要自己注册这些通知的发送方 —— 系统框架帮你发了，你只需要 addObserver。

### 3. Delegate + 手动刷新 — 最常用的一对一模式

你阶段 1 已经学了 Delegate 传值。在实际业务中，这其实是**最常用的"状态管理"方式**：数据变了，手动调一下 UI 更新。

```objc
// === 数据层（ViewModel 或 Model）===
@interface FeedViewModel : NSObject
@property (nonatomic, copy) NSArray *items;  // 列表数据
@end

@implementation FeedViewModel

- (void)fetchFeed {
    // 网络请求回来后
    self.items = responseObject[@"list"];
    // 通知 VC 刷新
    [self.delegate feedViewModelDidUpdate:self];
}
@end

// === UI 层（ViewController）===
- (void)feedViewModelDidUpdate:(FeedViewModel *)vm {
    // 手动刷新 UI —— 最朴素但也最常见
    [self.tableView reloadData];
}
```

**为什么这是最常用的？** 因为 ObjC/UIKit 里没有"声明式绑定"，绝大多数场景就是：数据改了 → 调 `reloadData` / `setText` / `setImage`。Delegate 只是告诉 VC "该刷了"，具体刷哪里还是你手动写。

## MVVM — ObjC 里怎么做"状态管理"

前面三个是"底层机制"，MVVM 是"上层架构"——告诉你 Model、ViewModel、View 之间谁管数据、谁管 UI、谁管绑定。

### C++ 类比

MVVM 和 Qt 的 **Document-View** / **Model-View** 是同一路思路：

```
C++ Qt:   Model (数据) ←→ View (展示) ←→ Controller (协调)
iOS ObjC: Model (数据) ←→ ViewModel (状态+逻辑) ←→ ViewController (UI)
```

### ObjC MVVM 的典型结构

```objc
// === Model — 纯数据，不含任何 UI 逻辑 ===
@interface User : NSObject
@property (nonatomic, copy) NSString *name;
@property (nonatomic, copy) NSString *avatarURL;
@property (nonatomic, assign) BOOL isVIP;
@end

// === ViewModel — 持有 Model，暴露"UI 友好"的属性和方法 ===
@interface UserViewModel : NSObject
@property (nonatomic, weak) id<UserViewModelDelegate> delegate;
@property (nonatomic, copy, readonly) NSString *displayName;    // "VIP·张三" 或 "张三"
@property (nonatomic, copy, readonly) NSString *avatarURLString;
@end

@protocol UserViewModelDelegate <NSObject>
- (void)userViewModelDidUpdate:(UserViewModel *)vm;
@end

@implementation UserViewModel

- (instancetype)initWithUser:(User *)user {
    self = [super init];
    if (self) {
        _user = user;
        // KVO 观察 user.name 变化 → 重新计算 displayName
        [user addObserver:self forKeyPath:@"name" options:0 context:nil];
    }
    return self;
}

- (NSString *)displayName {
    if (self.user.isVIP) {
        return [NSString stringWithFormat:@"VIP·%@", self.user.name];
    }
    return self.user.name;
}

- (void)observeValueForKeyPath:(NSString *)keyPath ofObject:(id)object change:(NSDictionary *)change context:(void *)context {
    // Model 变了 → 通知 VC
    [self.delegate userViewModelDidUpdate:self];
}
@end

// === ViewController — 只管 UI，不碰 Model ===
@interface ProfileVC : UIViewController <UserViewModelDelegate>
@property (nonatomic, strong) UserViewModel *viewModel;
@end

@implementation ProfileVC

- (void)viewDidLoad {
    [super viewDidLoad];
    self.viewModel.delegate = self;
    [self bindUI];  // 初始绑定
}

- (void)bindUI {
    self.nameLabel.text = self.viewModel.displayName;
    // 加载头像用 SDWebImage（下一篇网络通信再讲）
}

- (void)userViewModelDidUpdate:(UserViewModel *)vm {
    [self bindUI];  // 重新绑定 = 刷新
}
@end
```

### MVVM 的核心规则

| 规则 | 说明 | 违反的后果 |
|------|------|-----------|
| VC 不持有 Model | VC 只和 ViewModel 交互 | VC 里到处 `model.name`、`model.score`，和 Model 耦合，换数据源就要改 VC |
| ViewModel 不引用 UIKit | `#import <UIKit.h>` 不该出现在 VM 里 | VM 不可单元测试，失去了 MVVM 最大优势 |
| 绑定是单向的 | Model → ViewModel → VC | 双向绑定在 ObjC 里手动实现很痛苦，单向+回调更实际 |

### ObjC MVVM vs SwiftUI MVVM

| | ObjC MVVM | SwiftUI MVVM |
|---|-----------|--------------|
| 绑定方式 | 手动：Delegate/KVO/Block | 自动：`@Published` + `@ObservedObject` |
| UI 刷新 | 手动调 `reloadData` / `setText` | 自动：body 重新求值 |
| VM 可测试 | ✅ 可脱离 UIKit 单元测试 | ✅ 同理 |
| 代码量 | 多（样板代码） | 少 |

> ObjC 的 MVVM 本质是**手动 MVVM** —— 没有框架帮你自动绑定，你需要自己写"数据变了→通知→刷新"这条链路。

## 列表刷新的坑 — UITableView 的状态管理

列表是 iOS 中最频繁的状态管理场景。数据变了，怎么刷？

### reloadData vs insertRows

```objc
// 方式一：全量刷新（简单粗暴，但会丢失滚动位置）
- (void)feedViewModelDidUpdate:(FeedViewModel *)vm {
    self.items = vm.items;       // 更新数据源
    [self.tableView reloadData]; // 全量刷
}

// 方式二：增量刷新（保留滚动位置，动画流畅）
- (void)feedViewModelDidInsertItems:(FeedViewModel *)vm atIndexes:(NSIndexSet *)indexes {
    [self.tableView beginUpdates];
    [self.tableView insertRowsAtIndexPaths:indexPaths
                          withRowAnimation:UITableViewRowAnimationAutomatic];
    [self.tableView endUpdates];
}
```

### ⚠️ 增量刷新的 crash 陷阱

增量刷新要求 **数据源和 UI 严格同步**。调用 `insertRows` 前，数据源必须已经包含新数据，否则 crash：

```objc
// ❌ crash：数据源还没加，就告诉 tableView "多了一行"
[self.tableView insertRowsAtIndexPaths:@[newIndexPath]
                      withRowAnimation:UITableViewRowAnimationAutomatic];
// 此时 tableView 问 dataSource："第 N 行是什么？"  → 你还没加 → crash

// ✅ 正确：先改数据，再刷 UI
[self.items addObject:newItem];       // 1. 先改数据
[self.tableView insertRowsAtIndexPaths:@[newIndexPath]  // 2. 再通知 UI
                      withRowAnimation:UITableViewRowAnimationAutomatic];
```

> **原则**：先改 Model，再通知 View。和 C++ MVC/Qt Model-View 是同一个道理。

## 实战模式总结

你日常写业务时，99% 的场景用下面三个模式就够了：

### 模式一：简单页面 — VC 直接管 Model + 手动刷新

```objc
- (void)fetchData {
    [API requestFeed:^(NSArray *items) {
        self.items = items;         // 改数据
        [self.tableView reloadData]; // 刷 UI
    }];
}
```

**适用**：单页面、数据不需要跨页面共享。

### 模式二：需要跨页面共享 — ViewModel + Delegate

```objc
// AppDelegate 或某个全局 Service 持有 ViewModel
// 多个 VC 观察 ViewModel 的变化
```

**适用**：用户信息、购物车数量等全局状态。

### 模式三：全局事件 — NSNotification

```objc
// 登录成功 → 10 个页面都需要知道 → NSNotification
[[NSNotificationCenter defaultCenter] postNotificationName:@"UserDidLogin" object:nil];
```

**适用**：一对多、跨层级通知。

### 我理解到的核心 ✅必填

iOS 里"数据变了 UI 怎么跟着变"有三种底层机制。KVO 是属性级观察者，运行时动态创建子类来 hook setter，属性通过 setter 修改时会自动通知观察者，但必须手动 removeObserver 否则 crash，而且直接改 ivar（_score）不会触发。NSNotification 是全局广播，发收双方完全解耦，系统自己的通知（如前后台切换）是你日常最常接触的用例。Delegate + 手动刷新是最朴素也最常用的方式——数据改了手动调 reloadData 或 setText。上层架构方面，ObjC 的 MVVM 是"手动 MVVM"，没有框架帮你自动绑定，核心规则是 VC 不碰 Model、ViewModel 不引用 UIKit。列表刷新时最容易犯的错是先通知 UI 再改数据源，UITableView 会 crash，必须先改 Model 再通知 View。
