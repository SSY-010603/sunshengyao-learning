# 07 — 数据持久化

> 本文覆盖公共概念：⑦ 数据持久化

网络请求拿到的数据，App 重启后就没了。用户设置、缓存列表、本地草稿这些需要"存下来下次还在"的数据，就是数据持久化的范畴。

---

## 一句话定义

**数据持久化** = 把数据写到磁盘，App 重启后还能读回来。

---

## C++ 直觉

```
C++ 里你大概率用过：
- 配置文件：ini / json 文件读写  → 对应 iOS UserDefaults
- SQLite 数据库               → 对应 iOS SQLite / Core Data / GRDB
- 文件 I/O：fopen/fwrite      → 对应 iOS NSFileManager + 沙盒目录
- 内存缓存：LRU Cache          → 对应 iOS NSCache
```

iOS 有四层持久化方案，按复杂度递增：

| 层级 | 方案 | C++ 类比 | 适用场景 |
|------|------|----------|----------|
| 1 | UserDefaults | ini 文件 | 少量键值配置 |
| 2 | 文件存储（沙盒目录） | fread/fwrite | 图片缓存、下载文件 |
| 3 | SQLite / GRDB | SQLite3 C API | 结构化数据、列表缓存 |
| 4 | Core Data | ORM 框架 | 复杂关系模型（不推荐新项目用） |

## 第一层：UserDefaults — 少量键值存储

### 基本用法

```objc
// 写
[[NSUserDefaults standardUserDefaults] setBool:YES forKey:@"hasSeenOnboarding"];
[[NSUserDefaults standardUserDefaults] setInteger:30 forKey:@"fontSize"];
[[NSUserDefaults standardUserDefaults] setObject:@"dark" forKey:@"theme"];
[[NSUserDefaults standardUserDefaults] synchronize]; // 立即写盘（iOS 12+ 不必手动调，系统会择机写）

// 读
BOOL hasSeen = [[NSUserDefaults standardUserDefaults] boolForKey:@"hasSeenOnboarding"];
NSInteger fontSize = [[NSUserDefaults standardUserDefaults] integerForKey:@"fontSize"];
NSString *theme = [[NSUserDefaults standardUserDefaults] stringForKey:@"theme"];

// 删
[[NSUserDefaults standardUserDefaults] removeObjectForKey:@"hasSeenOnboarding"];
```

### ⚠️ UserDefaults 的坑

**坑 1：不是立即写盘**。`setObject:forKey:` 之后数据在内存里，系统会找时机异步写盘。如果 App 在写盘前 crash，数据丢失。关键数据调一下 `synchronize` 保底。

**坑 2：类型混淆**。存的是 bool，读的用了 `objectForKey`，拿到的是 NSNumber 不是 BOOL：

```objc
// 存
[[NSUserDefaults standardUserDefaults] setBool:YES forKey:@"flag"];

// 读法一 ✅ 正确
BOOL flag = [[NSUserDefaults standardUserDefaults] boolForKey:@"flag"];

// 读法二 ❌ 类型不对
id obj = [[NSUserDefaults standardUserDefaults] objectForKey:@"flag"];
// obj 是 NSNumber，不是 @YES 也不是 nil
// 如果你对它做 isEqualToString → crash
```

**坑 3：不适合存大数据**。UserDefaults 的 plist 文件是整体读写的，你存一个 1MB 的数组，每次启动都全量加载到内存。大数据用文件或数据库。

**坑 4：Suite 命名空间**。如果 App 有 Extension（比如 Share Extension、Widget），需要用 App Group 的 suite 来共享数据：

```objc
NSUserDefaults *shared = [[NSUserDefaults alloc] initWithSuiteName:@"group.com.app.shared"];
[shared setObject:@"hello" forKey:@"widgetTitle"];
```

## 第二层：文件存储 — 沙盒目录

iOS App 的文件系统叫**沙盒（Sandbox）**，每个 App 只能访问自己的目录，不能跨 App 读写。

### 沙盒目录结构

```
App 沙盒根目录/
├── Documents/       ← 持久化用户数据（会备份到 iCloud）
├── Library/
│   ├── Caches/      ← 缓存数据（系统磁盘不足时可自动清理，不会备份）
│   └── Preferences/ ← UserDefaults 的 plist 存在这里
└── tmp/             ← 临时文件（系统重启后可能被清理）
```

| 目录 | 何时清理 | iCloud 备份 | 用途 |
|------|----------|------------|------|
| Documents | 不会 | ✅ 会 | 用户主动保存的数据 |
| Library/Caches | 系统磁盘不足时 | ❌ 不会 | 网络图片缓存、接口数据缓存 |
| tmp | 重启后可能 | ❌ 不会 | 压缩/解压的临时文件 |

### 获取目录路径

```objc
// Documents
NSString *docDir = [NSSearchPathForDirectoriesInDomains(NSDocumentDirectory, NSUserDomainMask, YES) firstObject];

// Library/Caches
NSString *cacheDir = [NSSearchPathForDirectoriesInDomains(NSCachesDirectory, NSUserDomainMask, YES) firstObject];

// tmp
NSString *tmpDir = NSTemporaryDirectory();
```

### 文件读写

```objc
// 写文件
NSData *imageData = UIImageJPEGRepresentation(image, 0.8);
NSString *filePath = [docDir stringByAppendingPathComponent:@"avatar.jpg"];
[imageData writeToFile:filePath atomically:YES]; // atomically=YES 先写临时文件再 rename，防写一半 crash

// 读文件
NSData *data = [NSData dataWithContentsOfFile:filePath];
UIImage *img = [UIImage imageWithData:data];

// 判断文件是否存在
NSFileManager *fm = [NSFileManager defaultManager];
BOOL exists = [fm fileExistsAtPath:filePath];

// 删除文件
[fm removeItemAtPath:filePath error:nil];

// 计算缓存大小
NSURL *cacheURL = [NSURL fileURLWithPath:cacheDir];
NSDirectoryEnumerator *enumerator = [fm enumeratorAtURL:cacheURL
                              includingPropertiesForKeys:@[NSURLFileSizeKey]
                                                 options:0
                                            errorHandler:nil];
NSUInteger totalSize = 0;
for (NSURL *fileURL in enumerator) {
    NSNumber *fileSize;
    [fileURL getResourceValue:&fileSize forKey:NSURLFileSizeKey error:nil];
    totalSize += fileSize.unsignedIntegerValue;
}
// totalSize 单位是字节，除以 1024/1024 得 MB
```

### NSCache — 内存缓存

NSCache 类似 C++ 的 LRU Cache，系统内存不足时自动清理：

```objc
@interface FeedVC ()
@property (nonatomic, strong) NSCache<NSString *, UIImage *> *imageCache;
@end

- (NSCache *)imageCache {
    if (!_imageCache) {
        _imageCache = [[NSCache alloc] init];
        _imageCache.countLimit = 50;    // 最多 50 张
        _imageCache.totalCostLimit = 50 * 1024 * 1024; // 最多 50MB
    }
    return _imageCache;
}

// 存
[self.imageCache setObject:img forKey:coverURL];

// 取
UIImage *cached = [self.imageCache objectForKey:coverURL];
if (cached) {
    self.imageView.image = cached;
}
```

> NSCache 是内存缓存（App 杀掉就没了），SDWebImage 的磁盘缓存是文件缓存（持久化）。两者互补。

## 第三层：SQLite / GRDB — 结构化数据

### 为什么需要数据库？

UserDefaults 只能存键值对，文件只能存整块数据。当你需要：
- 按条件查询（"未读的消息"）
- 排序（"按时间倒序"）
- 关联（"这个用户发的所有帖子"）

就需要数据库。

### 系统自带的 SQLite

iOS 自带 SQLite3 C API，你直接 `#import <sqlite3.h>` 就能用。但 C API 太繁琐，日常开发几乎不用它直接写。

### GRDB — ObjC 项目的最佳选择

GRDB 是 Swift 写的 SQLite 封装库，但它的核心 API 可以从 ObjC 调用。如果你的项目是纯 ObjC 且不想引入 Swift 依赖，有两个选择：

1. **FMDB** — 纯 ObjC 的 SQLite 封装，老项目大量使用
2. **直接用 sqlite3 C API** — 如果查询简单

### FMDB 基本用法

```objc
#import "FMDB.h"

// 1. 打开数据库（文件不存在会自动创建）
NSString *dbPath = [docDir stringByAppendingPathComponent:@"app.db"];
FMDatabase *db = [FMDatabase databaseWithPath:dbPath];
if (![db open]) {
    NSLog(@"数据库打开失败");
    return;
}

// 2. 建表
[db executeUpdate:@"CREATE TABLE IF NOT EXISTS feed_item ("
                @"id INTEGER PRIMARY KEY, "
                @"title TEXT, "
                @"cover_url TEXT, "
                @"like_count INTEGER DEFAULT 0, "
                @"created_at REAL)"];

// 3. 插入
[db executeUpdate:@"INSERT INTO feed_item (title, cover_url, like_count, created_at) VALUES (?, ?, ?, ?)",
    item.title, item.coverURL, @(item.likeCount), @(NSDate.date.timeIntervalSince1970)];
// ⚠️ 用 ? 占位符，不要字符串拼接——防 SQL 注入

// 4. 查询
FMResultSet *rs = [db executeQuery:@"SELECT * FROM feed_item WHERE like_count > ? ORDER BY created_at DESC", @(10)];
while ([rs next]) {
    NSString *title = [rs stringForColumn:@"title"];
    NSInteger likes = [rs intForColumn:@"like_count"];
    // 构造 model...
}
[rs close];

// 5. 事务（多条操作必须原子性）
[db beginTransaction];
[db executeUpdate:@"UPDATE feed_item SET like_count = like_count + 1 WHERE id = ?", @(itemId)];
[db executeUpdate:@"INSERT INTO user_like (user_id, item_id) VALUES (?, ?)", @(userId), @(itemId)];
[db commit];
// 如果中途 crash，事务自动回滚
```

### ⚠️ 数据库的坑

**坑 1：线程安全**。FMDatabase 实例不能跨线程使用。多线程场景用 `FMDatabaseQueue`：

```objc
FMDatabaseQueue *queue = [FMDatabaseQueue databaseQueueWithPath:dbPath];
[queue inDatabase:^(FMDatabase *db) {
    // 所有操作在这个 block 里，保证线程安全
    [db executeUpdate:@"INSERT INTO ..."];
}];
[queue inTransaction:^(FMDatabase *db, BOOL *rollback) {
    [db executeUpdate:@"UPDATE ..."];
    [db executeUpdate:@"INSERT ..."];
    // 如果出错，设置 *rollback = YES 回滚
}];
```

**坑 2：数据库迁移**。App 迭代过程中表结构会变（加字段、加表），需要在 `open` 后执行 `ALTER TABLE`：

```objc
// 获取当前版本
NSInteger version = [db intForQuery:@"PRAGMA user_version"];
if (version < 2) {
    [db executeUpdate:@"ALTER TABLE feed_item ADD COLUMN is_liked INTEGER DEFAULT 0"];
    [db executeUpdate:@"PRAGMA user_version = 2"];
}
```

**坑 3：WAL 模式**。SQLite 默认用 journal 模式，并发读写性能差。开启 WAL (Write-Ahead Logging) 提升并发读：

```objc
[db executeUpdate:@"PRAGMA journal_mode=WAL"];
```

## 第四层：Core Data（了解即可）

Core Data 是 Apple 提供的 ORM 框架，类似 C++ 的 ODB 或 Java 的 Hibernate。

**为什么不推荐新项目用 Core Data？**
- API 复杂，学习成本高
- 多线程需要用 `NSManagedObjectContext` 的 `performBlock`，规则繁琐
- 迁移机制笨重（轻量迁移覆盖不了的场景要手动写 Mapping Model）
- 调试困难（SQL 日志要开参数才能看）

如果你的项目已经在用 Core Data，照着维护就行；新功能优先考虑 FMDB 或 GRDB。

## 缓存策略总结

| 数据类型 | 存哪 | 何时清理 | 示例 |
|----------|------|----------|------|
| 用户偏好 | UserDefaults | 用户手动重置 | 主题、字号、是否看过引导 |
| 网络图片 | Library/Caches + NSCache | 系统磁盘不足时 | 头像、封面图 |
| 接口数据缓存 | Library/Caches (文件) | 设置过期时间 / 系统清理 | 首页 feed 缓存 |
| 本地草稿 | Documents (数据库) | 用户手动删 | 未发送的评论、编辑中的文章 |
| 用户主数据 | Documents (数据库) | 永远不清理 | 聊天记录、收藏列表 |

**07.md 总结**：

iOS 数据持久化分四层。第一层 UserDefaults 存少量键值配置，类似 C++ 的 ini 文件，但要注意它不是立即写盘的、类型要匹配读写方法、不能存大数据（整个 plist 全量加载）。第二层是沙盒目录文件存储，Documents 放持久化用户数据会备份到 iCloud，Library/Caches 放缓存数据系统磁盘不足时可自动清理，tmp 放临时文件重启后可能被清理。NSCache 是内存级 LRU 缓存，设了 countLimit 和 totalCostLimit 后系统内存不足会自动淘汰。第三层 SQLite，日常 ObjC 项目用 FMDB 封装，占位符防 SQL 注入、FMDatabaseQueue 保证线程安全、事务保证原子性、PRAGMA user_version 做数据库迁移。第四层 Core Data 是 Apple 的 ORM 框架，API 复杂迁移笨重调试困难，新项目不建议用。选存储方案的关键是看数据类型：用户偏好用 UserDefaults、图片和接口缓存用 Caches 目录、本地草稿和用户主数据用数据库。
