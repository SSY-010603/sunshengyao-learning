# 08 — 多线程与异步编程

> 本文覆盖公共概念：⑧ 多线程与异步编程

你之前学的网络请求、数据库操作都是耗时的——如果这些操作在主线程跑，UI 就卡死。这篇讲 iOS ObjC 里怎么把耗时任务放到后台、完成后回主线程更新 UI，以及常见的线程陷阱。

---

## 一句话定义

**多线程与异步编程** = 把耗时任务放到后台线程执行，完成后回主线程更新 UI，同时避免线程安全问题。

---

## C++ 直觉

```
C++ 多线程：std::thread / std::async / std::future
C++ 线程池：自己实现 or 用第三方库
C++ 回主线程：没有"主线程"概念，GUI 框架各自实现（Qt 用信号槽，MFC 用 PostMessage）

iOS ObjC 多线程：GCD (Grand Central Dispatch)
iOS 线程池：GCD 内部管理，你不用自己创建线程
iOS 回主线程：DispatchQueue.main.async（和 Qt 的 QMetaObject::invokeMethod 类似）
```

GCD 是 Apple 在 OS 层面实现的线程池调度器，你只需要告诉它"做什么"和"在哪做"，不用管线程创建销毁。

---

## 主线程的铁律

iOS 里**所有 UI 操作必须在主线程**。这条规则没有例外。

```objc
// ❌ 在后台线程更新 UI → 可能 crash，可能不刷新，可能闪烁
dispatch_async(dispatch_get_global_queue(0, 0), ^{
    self.label.text = @"hello"; // 危险！
});

// ✅ 回主线程更新
dispatch_async(dispatch_get_global_queue(0, 0), ^{
    // 后台做耗时操作
    NSData *data = [NSURLConnection sendSynchronousRequest:request returningResponse:nil error:nil];
    
    dispatch_async(dispatch_get_main_queue(), ^{
        // 主线程刷 UI
        self.imageView.image = [UIImage imageWithData:data];
    });
});
```

UIKit 有 `assertMainThread` 检查，大部分 UI 方法在后台线程调用会直接 crash 或行为异常。UIKit 的文档里每个方法都标注了 "Must be called from the main thread"。

## GCD 核心概念

### 两种队列

| 队列类型 | 说明 | 创建方式 |
|----------|------|----------|
| 串行队列 (Serial) | 任务按顺序执行，一个完了一个才开始 | `dispatch_queue_create("com.app.task", DISPATCH_QUEUE_SERIAL)` |
| 并发队列 (Concurrent) | 任务同时执行，不保证完成顺序 | `dispatch_queue_create("com.app.task", DISPATCH_QUEUE_CONCURRENT)` |

### 系统预置队列

| 队列 | 类型 | 用途 |
|------|------|------|
| `dispatch_get_main_queue()` | 串行 | UI 更新 |
| `dispatch_get_global_queue(QOS_CLASS_DEFAULT, 0)` | 并发 | 普通后台任务 |
| `dispatch_get_global_queue(QOS_CLASS_USER_INITIATED, 0)` | 并发 | 用户触发、需要快返回 |
| `dispatch_get_global_queue(QOS_CLASS_UTILITY, 0)` | 并发 | 耗时计算、文件 I/O |
| `dispatch_get_global_queue(QOS_CLASS_BACKGROUND, 0)` | 并发 | 低优先级后台任务 |

QoS (Quality of Service) 类似 C++ 线程的优先级（nice 值），系统会根据 QoS 分配 CPU 时间和调度顺序。

### sync vs async

| 方法 | 说明 | 风险 |
|------|------|------|
| `dispatch_async(queue, block)` | 异步提交，不等待完成 | 安全 |
| `dispatch_sync(queue, block)` | 同步提交，阻塞当前线程等完成 | ⚠️ 容易死锁 |

```objc
// ❌ 死锁典型场景：在主线程 sync 到主队列
dispatch_sync(dispatch_get_main_queue(), ^{
    // 当前线程（主线程）等这个 block 执行完
    // 但这个 block 要在主队列（主线程）上执行
    // 主线程在等 block，block 在等主线程 → 死锁
    self.label.text = @"deadlock";
});
// 代码执行不到这里

// ✅ 正确：用 async
dispatch_async(dispatch_get_main_queue(), ^{
    self.label.text = @"safe";
});
```

**规则**：对当前队列调用 `dispatch_sync` = 死锁。**永远不要 sync 到自己所在的队列**。

## 常用 GCD 模式

### 模式一：后台干活 + 回主线程

这是你日常开发中最频繁的写法，占了 80% 的 GCD 使用场景：

```objc
- (void)loadUserData {
    dispatch_async(dispatch_get_global_queue(QOS_CLASS_UTILITY, 0), ^{
        // 后台：网络请求 / 文件读取 / 数据库查询
        NSDictionary *userData = [self fetchFromServer];
        
        dispatch_async(dispatch_get_main_queue(), ^{
            // 主线程：更新 UI
            self.nameLabel.text = userData[@"name"];
            self.avatarView.image = [UIImage imageWithData:userData[@"avatar"]];
        });
    });
}
```

### 模式二：dispatch_group — 等多个任务全部完成

C++ 类比：`std::latch` 或计数器 + condition_variable，等 N 个任务都结束再做汇总。

```objc
// 同时请求用户信息和配置，两个都回来后再刷新 UI
- (void)loadAllData {
    dispatch_group_t group = dispatch_group_create();
    
    __block NSDictionary *userInfo = nil;
    __block NSDictionary *config = nil;
    
    // 任务 1
    dispatch_group_enter(group);
    [API fetchUserInfo:^(NSDictionary *data) {
        userInfo = data;
        dispatch_group_leave(group);
    }];
    
    // 任务 2
    dispatch_group_enter(group);
    [API fetchConfig:^(NSDictionary *data) {
        config = data;
        dispatch_group_leave(group);
    }];
    
    // 两个都完成后执行
    dispatch_group_notify(group, dispatch_get_main_queue(), ^{
        [self updateUIWithUser:userInfo config:config];
    });
}
```

**`enter` 和 `leave` 必须配对**。多一个 leave → crash，少一个 leave → 永远不会 notify。

### 模式三：dispatch_after — 延迟执行

```objc
// 2 秒后执行
dispatch_after(dispatch_time(DISPATCH_TIME_NOW, (int64_t)(2.0 * NSEC_PER_SEC)),
               dispatch_get_main_queue(), ^{
    [self hideLoadingView];
});
```

⚠️ 注意 `dispatch_after` 不是精确定时器。它只是"至少等 N 秒后尽快执行"，如果主线程很忙，实际执行时间会延后。精确计时用 `NSTimer` 或 `CADisplayLink`。

### 模式四：dispatch_once — 线程安全的单例

```objc
// C++ 里你用 std::call_once + static 变量，ObjC 用 dispatch_once
+ (instancetype)sharedManager {
    static dispatch_once_t onceToken;
    static NetworkManager *instance;
    dispatch_once(&onceToken, ^{
        instance = [[NetworkManager alloc] init];
    });
    return instance;
}
```

`dispatch_once` 保证即使多线程同时调用，初始化代码也只执行一次。这是 ObjC 单例的标准写法。

### 模式五：dispatch_barrier — 读写锁

并发队列下"多读单写"场景：读可以并发，写必须独占。

```objc
// 并发队列
dispatch_queue_t queue = dispatch_queue_create("com.app.cache", DISPATCH_QUEUE_CONCURRENT);

// 读（可并发）
- (id)objectForKey:(NSString *)key {
    __block id value = nil;
    dispatch_sync(queue, ^{
        value = self.dict[key];
    });
    return value;
}

// 写（独占——barrier 阻塞其他任务直到写完）
- (void)setObject:(id)obj forKey:(NSString *)key {
    dispatch_barrier_async(queue, ^{
        self.dict[key] = obj;
    });
}
```

`dispatch_barrier_async` 的效果：等队列中已有任务全部完成 → 独占执行 barrier block → 完成后恢复并发。类似 C++ 的 `std::shared_mutex`（读用 `shared_lock`，写用 `unique_lock`）。

## OperationQueue — GCD 的面向对象封装

GCD 是 C 函数接口，OperationQueue 是 ObjC 面向对象的上层封装：

| | GCD | OperationQueue |
|---|-----|----------------|
| 接口 | C 函数 | ObjC 对象 |
| 取消任务 | 不支持（GCD 任务一旦提交无法取消） | `operation.cancel` |
| 任务依赖 | 不支持 | `op1 addDependency:op2`（op2 完成后才执行 op1） |
| 并发数控制 | 不支持 | `maxConcurrentOperationCount = 3` |
| 任务优先级 | QoS | `operation.queuePriority` |

```objc
// 使用 OperationQueue 做图片下载（控制并发数）
NSOperationQueue *downloadQueue = [[NSOperationQueue alloc] init];
downloadQueue.maxConcurrentOperationCount = 3; // 最多 3 个同时下载

for (FeedItem *item in items) {
    NSBlockOperation *op = [NSBlockOperation blockOperationWithBlock:^{
        NSData *data = [NSData dataWithContentsOfURL:[NSURL URLWithString:item.coverURL]];
        dispatch_async(dispatch_get_main_queue(), ^{
            item.coverImage = [UIImage imageWithData:data];
        });
    }];
    [downloadQueue addOperation:op];
}
```

**日常开发中选择**：简单场景用 GCD（轻量），需要取消/依赖/并发控制的用 OperationQueue。

## 线程安全 — 三个常见陷阱

### 陷阱一：竞态条件（Race Condition）

```objc
// ❌ 多线程同时修改同一个可变数组 → crash
// Thread A: [self.items addObject:newItem];
// Thread B: [self.items removeObjectAtIndex:0];
// 同时修改 NSMutableArray → 内部结构被破坏 → EXC_BAD_ACCESS

// ✅ 方案一：用串行队列保护
dispatch_queue_t safeQueue = dispatch_queue_create("com.app.items", DISPATCH_QUEUE_SERIAL);
dispatch_sync(safeQueue, ^{
    [self.items addObject:newItem];
});

// ✅ 方案二：用 @synchronized（简单但性能差，适合低频操作）
@synchronized(self.items) {
    [self.items addObject:newItem];
}
```

### 陷阱二：Block 捕获变量的时机

```objc
// ❌ 循环中 Block 捕获的是变量的引用，循环结束时 i 已经是 count
for (int i = 0; i < count; i++) {
    dispatch_async(queue, ^{
        [self processItem:i]; // i 永远是 count！
    });
}

// ✅ 用局部变量拷贝捕获当前值
for (int i = 0; i < count; i++) {
    int capturedI = i;
    dispatch_async(queue, ^{
        [self processItem:capturedI]; // 每次都是正确的 i
    });
}
```

### 陷阱三：异步回调中的 self 生命周期

```objc
// ❌ self 可能已经 dealloc
dispatch_async(queue, ^{
    [self doSomething]; // 如果 VC 已经 pop，self 是 zombie
});

// ✅ weak-strong dance
__weak typeof(self) weakSelf = self;
dispatch_async(queue, ^{
    __strong typeof(weakSelf) strongSelf = weakSelf;
    if (!strongSelf) return; // VC 已释放，直接返回
    [strongSelf doSomething];
});
```

你在 03.md（@property 与 ARC）和 crash/01-zombie.md 里已经学过 weak-strong dance，这里在异步回调场景下同样适用。后台线程的 Block 强引用了 self，导致 VC 无法 dealloc → 内存泄漏；或者 Block 执行时 self 已释放 → crash。

---

## 线程安全总结表

| 场景 | 安全做法 | 错误做法 |
|------|----------|----------|
| 多线程改可变集合 | 串行队列 / @synchronized / barrier | 直接改 |
| 循环中异步操作 | 局部变量捕获当前值 | Block 直接用循环变量 |
| 异步回调引用 self | weak-strong dance | 直接强引用 self |
| UI 更新 | 主线程 async | 后台线程改 UI |
| 对当前队列同步 | dispatch_async | dispatch_sync（死锁） |

**08.md 总结**：

iOS 多线程的核心铁律是所有 UI 操作必须在主线程，后台线程更新 UI 会 crash 或行为异常。GCD 是 Apple 在 OS 层面实现的线程池调度器，你只需要告诉它做什么和在哪个队列做。队列分串行（任务顺序执行）和并发（任务同时执行）两种，系统预置了主队列和不同 QoS 级别的全局并发队列。dispatch_sync 容易死锁——对当前队列调用 sync 会互相等待永远不返回，所以永远不要 sync 到自己所在的队列。日常开发最频繁的五种 GCD 模式：后台干活加回主线程刷 UI（占 80% 用法）、dispatch_group 等多个任务全部完成再汇总（enter 和 leave 必须配对）、dispatch_after 延迟执行（不是精确定时器）、dispatch_once 线程安全单例、dispatch_barrier 实现读写锁（多读单写）。OperationQueue 是 GCD 的面向对象封装，支持取消任务、任务依赖和并发数控制，需要这些能力时用它比 GCD 方便。三个常见线程陷阱：多线程同时修改可变集合会 crash（用串行队列或 @synchronized 保护）、循环中 Block 捕获循环变量拿到的是最终值（要用局部变量拷贝当前值）、异步回调中直接引用 self 可能导致内存泄漏或 crash（用 weak-strong dance）。
