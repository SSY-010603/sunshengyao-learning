# 09 — 权限管理

> 本文覆盖公共概念：⑨ 权限管理

阶段 1、2 你学的都是 App 内部的能力。但真实 App 要访问相机、相册、定位、通讯录、麦克风等敏感资源时，iOS 系统强制要求**先向用户申请授权**。这不仅是技术问题，更是 App Store 审核的红线——权限描述写错、时机不对，直接拒审。

---

## 一句话定义

**权限管理** = App 访问敏感资源（硬件、隐私数据）前，向系统声明用途、向用户申请授权、处理用户拒绝/撤销的完整流程。

---

## C++ 直觉

```
C++ 里：权限不是语言层面的事，是 OS 层面的——你 fopen "/etc/passwd" 失败返回 EACCES
        通常是服务器/守护进程的权限管理（chmod、ACL、SELinux）

iOS 里：权限是 App 开发的核心环节——Apple 强制要求 Info.plist 声明、运行时弹窗、用户可随时撤销
       所有权限申请都是异步的（用户决策需要时间），有完整的授权状态机
```

---

## 权限申请的四步流程

```
1. Info.plist 声明用途  ─→  没声明直接 crash，审核必拒
2. 检查当前授权状态      ─→  未决定/已授权/已拒绝/受限/未支持
3. 请求授权（弹窗）      ─→  系统弹窗，用户决策
4. 处理结果              ─→  授权后用，拒绝给降级方案或引导设置
```

**关键原则**：**先声明 → 再检查 → 后请求**。不能跳步。

## Info.plist 权限声明

### 为什么必须声明

iOS 10+ 开始，访问隐私资源的 App 必须在 Info.plist 里写明**用途描述**——这些描述会显示在系统弹窗里（"XXX 想要访问您的相机"）。没声明 → 调用 API 时直接 crash；声明了但文案不清晰 → App Store 审核拒绝。

### 常用权限键

| 权限 | Info.plist 键 | 示例值 |
|------|--------------|--------|
| 相机 | `NSCameraUsageDescription` | "用于拍摄照片和视频" |
| 相册读取 | `NSPhotoLibraryUsageDescription` | "用于选择照片上传" |
| 相册写入 | `NSPhotoLibraryAddUsageDescription` | "用于保存图片到相册" |
| 定位（使用时） | `NSLocationWhenInUseUsageDescription` | "用于显示您附近的商家" |
| 定位（始终） | `NSLocationAlwaysAndWhenInUseUsageDescription` | "用于后台记录您的运动轨迹" |
| 麦克风 | `NSMicrophoneUsageDescription` | "用于语音通话和录音" |
| 通讯录 | `NSContactsUsageDescription` | "用于查找通讯录中的好友" |
| 日历 | `NSCalendarsUsageDescription` | "用于添加日程提醒" |
| 提醒事项 | `NSRemindersUsageDescription` | "用于添加待办事项" |
| 健康 | `NSHealthShareUsageDescription` / `NSHealthUpdateUsageDescription` | "用于读取运动数据" |
| 蓝牙 | `NSBluetoothAlwaysUsageDescription` | "用于连接蓝牙设备" |
| 本地网络 | `NSLocalNetworkUsageDescription` | "用于发现局域网内的设备" |
| 跟踪 | `NSUserTrackingUsageDescription` | "用于为您提供更精准的广告推荐" |

### 用途描述的审核红线

Apple 审核对文案有严格要求：

```
❌ 不合格文案：
- "需要相机权限"  → 太模糊，没说用途
- "用于功能实现"  → 空话
- "授权后可使用相机"  → 仍是废话

✅ 合格文案：
- "用于拍摄商品照片上传至二手交易平台"
- "用于扫描二维码完成支付"
- "用于录制您练习口语的音频并打分"
```

**审核规则**：文案必须说清**具体用途**，不能只说"需要权限"。如果权限涉及多个用途，文案要覆盖主要场景。

## 三类授权状态机

iOS 权限 API 都有类似的四态状态机：

```
                  ┌──────────────┐
                  │  NotDetermined │ ← 从未请求过
                  └──────┬───────┘
                         │ request
                         ▼
              ┌──────────────────────┐
        ┌────>│    User-facing prompt    │<────┐
        │     └──────────┬───────────┘     │
        │                │                 │
        │   授权         │  拒绝           │
        ▼                ▼                 │
┌─────────────┐  ┌─────────────┐           │
│  Authorized  │  │   Denied    │           │
└─────────────┘  └─────────────┘           │
                                          │
              用户去设置改回                │
              ──────────────────────>──────┘
```

| 状态 | 含义 | 处理 |
|------|------|------|
| NotDetermined | 从未请求 | 调 request 触发系统弹窗 |
| Authorized | 已授权 | 正常使用 API |
| Denied | 用户拒绝 | 引导去设置开启 |
| Restricted | 系统限制（家长控制等） | 提示无法使用 |

## 相机权限实战

### 基本流程

```objc
#import <AVFoundation/AVFoundation.h>

- (void)checkAndRequestCamera {
    // 1. 检查当前状态
    AVAuthorizationStatus status = [AVCaptureDevice authorizationStatusForMediaType:AVMediaTypeVideo];
    
    switch (status) {
        case AVAuthorizationStatusNotDetermined: {
            // 2. 未决定 → 请求授权
            [AVCaptureDevice requestAccessForMediaType:AVMediaTypeVideo
                                       completionHandler:^(BOOL granted) {
                dispatch_async(dispatch_get_main_queue(), ^{
                    if (granted) {
                        [self startCamera];
                    } else {
                        [self showDeniedAlert];
                    }
                });
            }];
            break;
        }
        case AVAuthorizationStatusAuthorized: {
            // 3. 已授权 → 直接用
            [self startCamera];
            break;
        }
        case AVAuthorizationStatusDenied:
        case AVAuthorizationStatusRestricted: {
            // 4. 已拒绝 → 引导去设置
            [self showDeniedAlert];
            break;
        }
    }
}

- (void)showDeniedAlert {
    UIAlertController *alert = [UIAlertController alertControllerWithTitle:@"无法访问相机"
                                                                     message:@"请在设置中开启相机权限"
                                                              preferredStyle:UIAlertControllerStyleAlert];
    [alert addAction:[UIAlertAction actionWithTitle:@"去设置" style:UIAlertActionStyleDefault handler:^(UIAlertAction *action) {
        // 跳转到 App 的系统设置页
        NSURL *url = [NSURL URLWithString:UIApplicationOpenSettingsURLString];
        if ([[UIApplication sharedApplication] canOpenURL:url]) {
            [[UIApplication sharedApplication] openURL:url options:@{} completionHandler:nil];
        }
    }]];
    [alert addAction:[UIAlertAction actionWithTitle:@"取消" style:UIAlertActionStyleCancel handler:nil]];
    [self presentViewController:alert animated:YES completion:nil];
}
```

### ⚠️ 相机权限的坑

**坑 1：必须在主线程调起相机**。`AVCaptureSession *session = [[AVCaptureSession alloc] init]` 等操作涉及 UI，必须在主线程。

**坑 2：请求回调和用户决策异步**。`requestAccessForMediaType:completionHandler:` 的 completion 在**任意线程**，更新 UI 必须 dispatch 回主线程（你在 06.md 和 08.md 学过这条铁律）。

**坑 3：模拟器没相机**。模拟器上调起相机会失败，开发时要判断 `AVCaptureDevice` 是否存在。

## 相册权限实战

iOS 14 之后相册权限**分级**了，比相机复杂：

### PHAuthorizationStatus 四态

```objc
#import <Photos/Photos.h>

- (void)checkAndRequestPhotoLibrary {
    PHAuthorizationStatus status = [PHPhotoLibrary authorizationStatus];
    
    switch (status) {
        case PHAuthorizationStatusNotDetermined: {
            [PHPhotoLibrary requestAuthorization:^(PHAuthorizationStatus newStatus) {
                dispatch_async(dispatch_get_main_queue(), ^{
                    [self handlePhotoStatus:newStatus];
                });
            }];
            break;
        }
        default:
            [self handlePhotoStatus:status];
            break;
    }
}

- (void)handlePhotoStatus:(PHAuthorizationStatus)status {
    switch (status) {
        case PHAuthorizationStatusAuthorized:
            // 完全授权，可读可写
            [self showPhotoPicker];
            break;
        case PHAuthorizationStatusLimited:
            // ⚠️ iOS 14+ 新增：用户只授权了部分照片
            [self showLimitedPhotoPicker];
            break;
        case PHAuthorizationStatusDenied:
        case PHAuthorizationStatusRestricted:
            [self showDeniedAlert];
            break;
    }
}
```

### iOS 14+ 的 Limited 权限

用户可以只选部分照片授权给 App，App 只能看到这些照片。这是 Apple 的隐私增强功能。

```objc
// 检查是否是 limited 状态
if ([PHPhotoLibrary authorizationStatus] == PHAuthorizationStatusLimited) {
    // 展示"管理照片"入口，让用户可以重新选择
    [self showManagePhotosEntry];
}

// 让用户重新选择授权哪些照片
[[PHPhotoLibrary sharedPhotoLibrary] presentLimitedLibraryPickerFromViewController:self];
```

### 写入相册（不需要授权）

```objc
// iOS 14+ 用 NSPhotoLibraryAddUsageDescription 声明，写入相册可以只申请"添加"权限
[[PHPhotoLibrary sharedPhotoLibrary] performChanges:^{
    [PHAssetCreationRequest creationRequestForAssetFromImage:image];
} completionHandler:^(BOOL success, NSError *error) {
    dispatch_async(dispatch_get_main_queue(), ^{
        if (success) {
            [self showToast:@"保存成功"];
        } else {
            [self showToast:@"保存失败"];
        }
    });
}];
```

## 定位权限实战

定位是权限里最复杂的，因为涉及三种模式：

| 模式 | Info.plist 键 | 说明 |
|------|---------------|------|
| 使用时授权 | `NSLocationWhenInUseUsageDescription` | App 在前台时才定位 |
| 始终授权 | `NSLocationAlwaysAndWhenInUseUsageDescription` | 后台也能定位（如运动追踪） |
| 后台定位 | 后台模式 `UIBackgroundModes` 设 `location` + 始终授权 | App 被挂起后仍能收到位置更新 |

```objc
#import <CoreLocation/CoreLocation.h>

@interface LocationManager () <CLLocationManagerDelegate>
@property (nonatomic, strong) CLLocationManager *manager;
@end

@implementation LocationManager

- (instancetype)init {
    self = [super init];
    if (self) {
        self.manager = [[CLLocationManager alloc] init];
        self.manager.delegate = self;
        self.manager.desiredAccuracy = kCLLocationAccuracyBest;
    }
    return self;
}

- (void)requestLocation {
    CLAuthorizationStatus status = [self.manager authorizationStatus];
    
    switch (status) {
        case kCLAuthorizationStatusNotDetermined:
            // iOS 13+ 用这个方法（带 always 参数）
            if (@available(iOS 13.0, *)) {
                [self.manager requestWhenInUseAuthorization];  // 先请求使用时授权
            }
            break;
        case kCLAuthorizationStatusAuthorizedWhenInUse:
            // 已有使用时授权，如果要后台定位，再申请始终授权
            [self.manager requestAlwaysAuthorization];
            break;
        case kCLAuthorizationStatusDenied:
        case kCLAuthorizationStatusRestricted:
            [self showDeniedAlert];
            break;
        case kCLAuthorizationStatusAuthorizedAlways:
            [self.manager startUpdatingLocation];
            break;
    }
}

#pragma mark - CLLocationManagerDelegate

- (void)locationManager:(CLLocationManager *)manager didChangeAuthorizationStatus:(CLAuthorizationStatus)status {
    // ⚠️ 用户在设置里改了权限会触发这里
    if (status == kCLAuthorizationStatusAuthorizedWhenInUse ||
        status == kCLAuthorizationStatusAuthorizedAlways) {
        [manager startUpdatingLocation];
    } else if (status == kCLAuthorizationStatusDenied) {
        [manager stopUpdatingLocation];
        [self showDeniedAlert];
    }
}

- (void)locationManager:(CLLocationManager *)manager didUpdateLocations:(NSArray<CLLocation *> *)locations {
    CLLocation *location = locations.lastObject;
    NSLog(@"经纬度：%f, %f", location.coordinate.latitude, location.coordinate.longitude);
    [manager stopUpdatingLocation];  // 拿到一次就停，避免持续耗电
}

@end
```

### ⚠️ 定位权限的坑

**坑 1：请求时机**。不能在 `viewDidLoad` 直接请求（用户体验差），要在用户**明确需要定位的场景**请求——比如点击了"附近商家"按钮。

**坑 2：授权状态变化回调**。用户可能在系统设置里改了权限，App 要监听 `locationManagerDidChangeAuthorization:` (iOS 14+) 及时响应。

**坑 3：后台定位审核严**。App Store 对申请"始终授权 + 后台定位"的 App 审核极严，要明确说明为什么需要后台定位，否则审核拒绝。

## 跟踪权限（ATT 框架）

iOS 14.5+ 引入 **App Tracking Transparency (ATT)**，跨 App 跟踪用户必须先请求授权。这是广告行业影响最大的权限变更。

```objc
#import <AppTrackingTransparency/AppTrackingTransparency.h>
#import <AdSupport/AdSupport.h>

- (void)requestTrackingAuthorization {
    if (@available(iOS 14, *)) {
        ATTrackingManagerAuthorizationStatus status = [ATTrackingManager trackingAuthorizationStatus];
        
        if (status == ATTrackingManagerAuthorizationStatusNotDetermined) {
            [ATTrackingManager requestTrackingAuthorizationWithCompletionHandler:^(ATTrackingManagerAuthorizationStatus status) {
                dispatch_async(dispatch_get_main_queue(), ^{
                    if (status == ATTrackingManagerAuthorizationStatusAuthorized) {
                        NSString *idfa = [[ASIdentifierManager sharedManager] advertisingIdentifier].UUIDString;
                        // 上报 IDFA 给广告 SDK
                    } else {
                        // 用户拒绝跟踪，IDFA 返回全 0
                    }
                });
            }];
        }
    }
}
```

**注意**：ATT 弹窗必须在 App 启动后尽快弹（或在用户做需要跟踪的操作前），否则广告 SDK 拿不到 IDFA，广告收入会大幅下降。

## 权限的撤销与生命周期

### 用户可随时撤销权限

iOS 设置 → 隐私 → 对应权限 → 选中你的 App → 关闭。这是用户的基本权利，App 不能阻止。

### App 要监听权限变化

```objc
// 在 App 进入前台时重新检查权限（用户可能在后台改了设置）
- (void)applicationWillEnterForeground:(NSNotification *)note {
    [self recheckAllPermissions];
}

// 或监听 UIApplicationDidBecomeActiveNotification
[[NSNotificationCenter defaultCenter] addObserver:self
                                         selector:@selector(recheckAllPermissions)
                                             name:UIApplicationDidBecomeActiveNotification
                                           object:nil];
```

### 权限被撤销时的降级处理

```objc
- (void)handleCameraRevoked {
    // 1. 停止所有正在进行的相机操作
    [self.cameraSession stopRunning];
    
    // 2. 清理相关 UI 状态
    [self hideCameraButton];
    
    // 3. 显示降级方案（如：让用户从相册选）
    [self showPhotoLibraryAlternative];
}
```

## 权限的授权时机策略

### 推荐时机

| 时机 | 示例 | 优点 |
|------|------|------|
| 用户主动触发 | 点击"拍照"按钮 → 请求相机权限 | 用户有预期，授权率高 |
| 解释后再请求 | 先展示"为什么需要"页 → 用户点"我知道了" → 请求 | 授权率更高 |
| 首次进入功能页 | 首次打开"附近"页 → 请求定位 | 场景明确 |

### 不推荐时机

| 时机 | 问题 |
|------|------|
| App 启动时一次申请一堆 | 用户不知道每个权限干嘛的，授权率低；且像 ATT 这种可能被系统限制 |
| 用户没交互时弹 | 用户没预期，容易直接拒 |

### "预解释" 模式

在系统弹窗前，自己先弹一个说明页，让用户知道权限用途，然后再请求：

```objc
- (void)requestCameraWithPreExplanation {
    UIAlertController *preAlert = [UIAlertController
        alertControllerWithTitle:@"开启相机权限"
                           message:@"我们需要相机权限来帮您拍摄商品照片，不会拍摄您不希望公开的内容"
                    preferredStyle:UIAlertControllerStyleAlert];
    
    [preAlert addAction:[UIAlertAction actionWithTitle:@"开启" style:UIAlertActionStyleDefault handler:^(UIAlertAction *action) {
        // 用户点"开启"后再调真正的权限请求
        [self requestCameraPermission];
    }]];
    [preAlert addAction:[UIAlertAction actionWithTitle:@"暂不" style:UIAlertActionStyleCancel handler:nil]];
    
    [self presentViewController:preAlert animated:YES completion:nil];
}
```

这种模式能让授权率提升 30-50%，因为用户拒绝自己 App 的弹窗比拒绝系统弹窗心理成本低。

## 权限管理的常见审核拒绝原因

| 原因 | 解决 |
|------|------|
| Info.plist 没声明权限键 | 加上对应键值 |
| 用途描述模糊（如"需要权限"） | 写清具体用途 |
| 申请了权限但没用到 | 去掉 Info.plist 声明 |
| 后台定位申请理由不充分 | 明确说明"为什么需要后台定位" |
| ATT 弹窗时机不对 | 在用户做相关操作前请求 |
| 权限被拒绝后强制不让用 App | 提供降级方案或友好提示 |

**09.md 总结**：
iOS 权限管理四步流程是 Info.plist 声明用途 → 检查当前授权状态 → 请求授权触发系统弹窗 → 处理结果。Info.plist 声明是审核红线，没声明直接 crash，文案模糊（如"需要权限"）会被拒，必须写清具体用途（如"用于拍摄商品照片上传"）。所有权限都是四态状态机：未决定（NotDetermined）、已授权（Authorized）、已拒绝（Denied）、受限（Restricted），处理逻辑要覆盖每个分支。相机的 API 是 `AVCaptureDevice requestAccessForMediaType:`，回调在任意线程需回主线程。相册权限 iOS 14+ 新增 Limited 模式（用户只授权部分照片），读写权限分离用不同的 Info.plist 键。定位权限有使用时（WhenInUse）和始终（Always）两级授权，后台定位审核极严必须说明充分理由。ATT 跟踪权限是 iOS 14.5+ 引入的，跨 App 跟踪用户前必须请求授权，IDFA 获取的前提。用户可随时在系统设置撤销权限，App 进入前台时要重新检查状态并提供降级方案。授权时机最忌在 App 启动时一次申请一堆，推荐用户主动触发时请求，配合"预解释"模式（自己先弹一个说明页再调系统弹窗）能让授权率提升 30-50%。
