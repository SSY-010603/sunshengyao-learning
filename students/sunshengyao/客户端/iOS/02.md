## 从上一篇到这里

上一篇你学了 UIView 和视图层级——界面是"怎么搭出来的"。这一篇解决的是另一个问题：**界面从出现到消失，经历了哪些阶段？在每个阶段你该做什么？**

这就像 C++ 中对象的生命周期：构造函数里初始化成员变量，析构函数里释放资源。iOS 的 UIViewController 也有类似的"构造-使用-析构"流程，只不过它被拆成了更多的回调节点，让你可以更精细地控制每个阶段的行为。

## 核心概念：UIViewController 的生命周期

UIViewController 是 iOS 中"一个页面"的代名词。你看到的每一个屏幕，背后几乎都是一个 UIViewController（简称 VC）。它从被创建到被销毁，会依次触发一系列回调方法——这就是**页面级生命周期**。

### 完整生命周期流程图

```
初始化 (init)
    │
    ▼
viewDidLoad          ← 只调用一次，视图刚加载完
    │
    ▼
viewWillAppear       ← 每次视图"即将出现"前调用
    │
    ▼
viewDidAppear        ← 视图已经出现在屏幕上
    │
    │   ┌──────────────────────┐
    │   │ 用户在页面上操作...     │
    │   │ 可能跳转到其他页面      │
    │   └──────────────────────┘
    │
    ▼
viewWillDisappear   ← 视图"即将消失"前调用
    │
    ▼
viewDidDisappear    ← 视图已经从屏幕上消失
    │
    │   ┌──────────────────────┐
    │   │ 可能又回到这个页面      │
    │   └──────────────────────┘
    │
    ▼
viewWillAppear       ← 再次出现
    │
    ▼
viewDidAppear
    │
    ...
    │
    ▼
deinit               ← VC 被释放，类似 C++ 析构函数
```

### C++ 类比

| iOS 生命周期 | C++ 类比 | 说明 |
|-------------|---------|------|
| `init` | 构造函数 | 分配内存，初始化属性 |
| `viewDidLoad` | 构造函数的"后处理" | 视图已加载，做一次性初始化 |
| `viewWillAppear` | — | C++ 没有对应物；这是 iOS 特有的"即将显示"通知 |
| `viewDidAppear` | — | 视图已可见，适合启动动画或定时器 |
| `viewWillDisappear` | — | 即将离开，适合暂停动画或定时器 |
| `viewDidDisappear` | — | 已不可见，适合暂停耗时操作 |
| `deinit` | 析构函数 | 释放资源 |

> **关键区别**：C++ 对象的构造和析构通常各只调一次。但 iOS 的 VC 不同——`viewWillAppear` 和 `viewDidAppear` 可能被调用**多次**（每次页面从隐藏回到可见时都会触发）。这是因为 iOS 的导航是**栈式**的：你从 A 页面跳到 B 页面，A 不会被销毁，只是被盖住了；当你从 B 返回 A 时，A 再次变为可见，`viewWillAppear` 再次触发。

## 每个回调里该做什么？

这是实际开发中最核心的知识——**知道在哪个回调里放什么代码**。放错了地方，要么功能不生效，要么性能出问题。

### viewDidLoad — 一次性的初始化

```swift
override func viewDidLoad() {
    super.viewDidLoad()
    
    // ✅ 适合做的事：
    view.backgroundColor = .white
    setupUI()           // 创建和添加子视图
    setupConstraints()  // 设置 Auto Layout 约束
    loadData()          // 首次加载数据
    
    // ❌ 不适合做的事：
    // 不要在这里启动需要每次出现都刷新的逻辑
    // 不要假设视图已经在屏幕上（此时视图还不可见）
}
```

**C++ 直觉**：这就像在构造函数里初始化所有成员变量、分配资源——做一次，后续不用重复。

### viewWillAppear — 每次出现前的准备

```swift
override func viewWillAppear(_ animated: Bool) {
    super.viewWillAppear(animated)
    
    // ✅ 适合做的事：
    updateNavigationBar()  // 刷新导航栏状态
    refreshData()          // 刷新数据（从其他页面返回时数据可能变了）
    startObserving()       // 注册通知监听
    
    // ❌ 不适合做的事：
    // 不要在这里做耗时的数据请求（会阻塞页面出现）
    // 不要在这里创建子视图（属于一次性操作，放 viewDidLoad）
}
```

### viewDidAppear — 页面已经可见

```swift
override func viewDidAppear(_ animated: Bool) {
    super.viewDidAppear(animated)
    
    // ✅ 适合做的事：
    startAnimation()       // 启动动画（视图必须可见才能看到）
    startTimer()           // 启动定时器
    trackPageView()        // 埋点统计（页面曝光）
    
    // ❌ 不适合做的事：
    // 不要在这里修改布局（视图已经显示，改布局会导致闪烁）
}
```

### viewWillDisappear / viewDidDisappear — 离开时的清理

```swift
override func viewWillDisappear(_ animated: Bool) {
    super.viewWillDisappear(animated)
    
    // ✅ 适合做的事：
    stopAnimation()        // 停止动画
    stopTimer()            // 停止定时器
    removeObserving()      // 移除通知监听
    saveDraft()            // 保存草稿数据
}

override func viewDidDisappear(_ animated: Bool) {
    super.viewDidDisappear(animated)
    
    // ✅ 适合做的事：
    cancelNetworkRequest() // 取消不需要的网络请求
    pauseVideo()           // 暂停视频播放
}
```

### deinit — 最终释放

```swift
deinit {
    // ✅ 必须做的事：
    NotificationCenter.default.removeObserver(self)  // 移除所有通知监听
    // 释放任何强引用的代理/回调
}
```

> **C++ 程序员的警觉**：在 C++ 中，析构函数里释放资源是自动的（RAII）。但在 iOS 中，如果你忘了在 `deinit` 或 `viewDidDisappear` 里清理监听器和定时器，它们会继续运行，造成**内存泄漏**或**野指针崩溃**。这不是 RAII，你得自己管。

## 实战：一个带生命周期的页面

把上一篇的 Hello 页面加上生命周期日志，你就能亲眼看到调用顺序：

```swift
import UIKit

class LifecycleViewController: UIViewController {
    
    let statusLabel = UILabel()
    
    override func viewDidLoad() {
        super.viewDidLoad()
        print("📄 viewDidLoad — 视图已加载，做一次性初始化")
        
        view.backgroundColor = .white
        
        statusLabel.font = UIFont.systemFont(ofSize: 18)
        statusLabel.textAlignment = .center
        statusLabel.text = "等待生命周期事件..."
        statusLabel.translatesAutoresizingMaskIntoConstraints = false
        view.addSubview(statusLabel)
        
        NSLayoutConstraint.activate([
            statusLabel.centerXAnchor.constraint(equalTo: view.centerXAnchor),
            statusLabel.centerYAnchor.constraint(equalTo: view.centerYAnchor)
        ])
    }
    
    override func viewWillAppear(_ animated: Bool) {
        super.viewWillAppear(animated)
        statusLabel.text = "viewWillAppear: 即将出现"
        print("📄 viewWillAppear — 即将出现在屏幕上")
    }
    
    override func viewDidAppear(_ animated: Bool) {
        super.viewDidAppear(animated)
        statusLabel.text = "viewDidAppear: 已经可见"
        print("📄 viewDidAppear — 已经出现在屏幕上")
    }
    
    override func viewWillDisappear(_ animated: Bool) {
        super.viewWillDisappear(animated)
        print("📄 viewWillDisappear — 即将离开屏幕")
    }
    
    override func viewDidDisappear(_ animated: Bool) {
        super.viewDidDisappear(animated)
        print("📄 viewDidDisappear — 已经离开屏幕")
    }
    
    deinit {
        print("📄 deinit — 控制器被释放")
    }
}
```

> **动手建议**：你可以把这个 VC 作为 App 的根控制器，然后从它跳转到另一个页面再返回，观察控制台输出。你会看到 `viewWillAppear/viewDidAppear` 被多次调用，而 `viewDidLoad` 只调用一次。

## App 级别生命周期

除了页面的生命周期，iOS 还有 **App 级别**的生命周期——整个应用进入前台、后台、被终止等：

```
App 启动
    │
    ▼
applicationDidFinishLaunching   ← App 启动完成，做全局初始化
    │
    ▼
applicationDidBecomeActive      ← App 获得焦点，用户可以交互
    │
    │   ┌──────────────────────┐
    │   │ 用户操作中...          │
    │   │ 电话来了 / 按了 Home   │
    │   └──────────────────────┘
    │
    ▼
applicationWillResignActive     ← App 即将失去焦点
    │
    ▼
applicationDidEnterBackground   ← App 进入后台
    │
    │   ┌──────────────────────┐
    │   │ 用户切回 App           │
    │   └──────────────────────┘
    │
    ▼
applicationWillEnterForeground  ← App 即将回到前台
    │
    ▼
applicationDidBecomeActive      ← 再次获得焦点
    │
    ...
    │
    ▼
applicationWillTerminate        ← App 即将被终止
```

### 在 AppDelegate / SceneDelegate 中处理

```swift
// AppDelegate.swift（传统方式）
func applicationDidBecomeActive(_ application: UIApplication) {
    // App 回到前台，恢复动画、刷新数据
}

func applicationDidEnterBackground(_ application: UIApplication) {
    // App 进入后台，保存数据、暂停任务
    // ⚠️ 你只有大约 5 秒时间做保存操作！
}
```

> **C++ 程序员的直觉**：App 生命周期就像操作系统的进程状态转换——运行→阻塞→就绪→终止。你在操作系统课上学过的状态机模型，在这里完全适用。

## 常见陷阱

| 陷阱 | 后果 | 正确做法 |
|------|------|----------|
| 在 `viewDidLoad` 里做每次都需要刷新的逻辑 | 从其他页面返回时数据不更新 | 放到 `viewWillAppear` |
| 在 `viewWillAppear` 里创建子视图 | 重复添加，导致视图叠加 | 创建放 `viewDidLoad`，刷新放 `viewWillAppear` |
| 在 `viewDidAppear` 里修改布局 | 视图闪烁 | 布局修改放 `viewDidLoad` 或 `viewWillAppear` |
| 忘了在 `deinit` 移除通知监听 | 野指针崩溃，收到通知时访问已释放的对象 | `deinit` 中 `removeObserver(self)` |
| 忘了在 `viewDidDisappear` 停止定时器 | 定时器在后台继续跑，浪费 CPU 和电量 | `viewDidDisappear` 中 `invalidate()` |
| 在 `applicationDidEnterBackground` 做耗时操作 | 系统直接杀掉 App | 后台只有约 5 秒，用 `beginBackgroundTask` 延长 |

## 关键要点总结

1. **UIViewController 生命周期**：`viewDidLoad`（一次性初始化）→ `viewWillAppear/viewDidAppear`（每次可见触发）→ `viewWillDisappear/viewDidDisappear`（每次离开触发）→ `deinit`（释放）
2. **核心区分**：`viewDidLoad` 只调一次，`viewWillAppear/viewDidAppear` 可能调多次——这是导航栈机制决定的
3. **每个回调有适合做的事**：初始化放 `viewDidLoad`，刷新放 `viewWillAppear`，动画/定时器放 `viewDidAppear`，清理放 `viewWillDisappear`/`deinit`
4. **App 生命周期**：前台→后台切换由 AppDelegate/SceneDelegate 处理，后台只有约 5 秒保存时间
5. **最大陷阱**：忘了清理监听器和定时器 → 内存泄漏或崩溃，C++ 的 RAII 不会帮你兜底
